# AI智能个人健康饮食助手 — 软件全面介绍

## 一、项目概述

**AI智能个人健康饮食助手**是一款面向普通用户的移动端健康饮食管理应用，作为杭州师范大学（HZNU）课程设计项目，由21小组4名成员共同完成。

系统通过**拍照识别、语音输入、手动编辑**三种方式采集用户的饮食数据，结合内置的食物营养成分库计算每日营养摄入，并根据用户个性化的健康目标（减脂、增肌、控糖等）生成**健康评分、改进建议、个性化食谱推荐和超标预警**。

## 三、技术架构

### 3.1 总体架构

系统采用**前后端分离的分层架构**：

```
移动端 UI（Vue 3 + Vite）
    ↓ RESTful JSON API
Spring Boot 后端
    ├── Controller 层    — 请求入口、参数校验、统一响应
    ├── Service 层       — 业务逻辑（记录、营养计算、评分、推荐、预警）
    ├── AI Adapter 层    — 适配第三方 AI 服务（模拟实现）
    └── Repository 层    — JPA 数据访问
    ↓
数据库（H2 / MySQL）
```

## 四、详细功能模块

### 模块 1：饮食数据采集（首页/记录页）

三种数据采集入口：

**① 拍照识别食物（UC-01）**
- 模拟拍照后调用图像识别，返回候选食物列表（含置信度）
- 用户从候选列表中选择确认，指定餐次和份量后保存
- 低置信度或未识别时提示用户改用手动添加

**② 语音输入饮食记录（UC-02）**
- 模拟语音录制，调用语音转文字 + 食物实体解析
- 示例："我中午吃了一碗米饭和一个鸡腿"
- 解析为：[米饭, 1碗, 午餐]、[鸡腿, 1个, 午餐]
- 用户可编辑修正解析结果后批量保存

**③ 手动添加（UC-03）**
- 表单填写：食物名称、餐次（早餐/午餐/晚餐/夜宵/其他）、份量
- 当 AI 识别失败或用户需要补记时使用

**记录管理（手风琴布局）：**
- 按餐次分组以手风琴卡片展示今日已记录的食物
- 折叠状态：显示餐次图标+名称、该餐热量/蛋白质合计
- 展开状态：顶部可滑动照片区（多张横向滑动），下方每条食物显示：
  - 食物名称、份量、来源标签
  - 热量/蛋白质/脂肪/碳水/糖/钠 6项营养值
  - 编辑/删除按钮
- 点击照片缩略图弹出全屏大图预览
- 顶部今日总览卡片：展示6项营养当日总和（热量/蛋白质/脂肪/碳水/糖/钠）

### 模块 2：智能营养分析（分析页）

**数据来源：** 饮食记录 × 食物营养成分库

**计算逻辑：**
```
单条记录营养 = 食物每100g营养 × (用户确认份量 / 100)
当日汇总     = 所有记录营养之和
```

**功能：**
- 以网格卡片展示热量/蛋白质/脂肪/碳水四大指标的摄入量、目标值、完成百分比
- 完整度按目标百分比计算（如热量摄入 1200/1600 = 75%）
- **近一周热量趋势柱状图**，方便用户观察饮食趋势
- 自动生成**饮食改进建议**（如"蛋白质摄入不足，建议增加豆制品"）

**目标体系：**
| 目标 | 热量 | 蛋白质 | 脂肪 | 碳水 |
|------|------|--------|------|------|
| 均衡 | 2000 | 60g | 65g | 300g |
| 减脂 | 1600 | 70g | 50g | 200g |
| 增肌 | 2500 | 120g | 70g | 350g |

### 模块 3：健康评分与建议（评分页）

**评分逻辑：**
- 基础分：80 分
- 热量超标：每超出目标 10% 扣 1 分（最多扣 10 分）
- 糖分 > 50g：扣 5 分
- 钠 > 2400mg：扣 5 分
- 蛋白质达标：加 5 分
- 最终范围：0-100 分

**前置条件：**
- 当日记录不足 2 餐 → 不评分，提示"记录更多饮食后可获得评分"

**输出内容：**
- 环形进度图展示评分
- ✅ 优点列表（如"热量控制良好"、"糖分控制良好"）
- ⚠️ 风险项列表（如"蛋白质摄入不足"）
- 💡 改进建议列表
- 📈 近一周评分趋势图

### 模块 4：个性化食谱推荐（推荐页）

**推荐算法（基于内容的评分排序）：**
1. **过滤阶段**：排除用户忌口的食谱、历史"不喜欢"的食谱
2. **评分阶段**（0-100分）：
   - 基础分：50
   - 目标匹配（±20分）：减脂偏好低热量食谱（<300kcal），增肌偏好高蛋白食谱（>20g）
   - 口味偏好匹配（+5/项）
   - 标签匹配（+15分）
   - 违反忌口（-50分，直接过滤）
3. **排序输出**：取 Top 3-5 条推荐

**推荐理由生成：**
根据匹配项动态生成，如：
> "推荐理由：这道菜低热量、低卡、清淡，符合您的减脂目标"

**交互：**
- 用户点击"不喜欢"后，系统记录反馈并即时替换为新菜品
- 同时检测饮食预警，在推荐页底部展示

### 模块 5：饮食提醒与预警（推荐页底 + 设置）

**预警检测时机：** 每次查看推荐页时自动检测

**检测逻辑：**
```
当日累计摄入 > 用户设置的阈值 → 生成预警消息
```

**预设阈值（演示用户）：**
| 指标 | 阈值 |
|------|------|
| 热量 | 2000 kcal |
| 糖分 | 50 g |
| 钠 | 2400 mg |

**预警消息示例：**
> "今日钠摄入 2850.0，已超过阈值 2400"
> "建议减少盐分摄入，多吃新鲜蔬果"

### 模块 6：健康档案与设置（我的页）

- 个人资料：年龄、身高、体重
- 健康目标：均衡饮食 / 减脂 / 增肌 / 控糖
- 口味偏好：多选（清淡、中式、西式、日式、辣味、酸甜、咸鲜）
- 忌口设置：多选（海鲜、花生、牛奶、鸡蛋、豆制品、辛辣）
- 慢性病/特殊饮食标签：糖尿病、高血压、高血脂、痛风
- 预警阈值：可分别设置热量/糖分/钠的每日上限，支持开关
- 所有设置保存后实时影响推荐算法和预警检测

## 五、数据库设计

系统共设计 **8 张核心数据表**：

| 表名 | 说明 | 核心字段 |
|------|------|---------|
| `users` | 用户基础信息 | id, nickname, created_at |
| `user_profile` | 用户健康档案 | user_id, age, height, weight, goal, taboo |
| `food_item` | 食物营养成分库（每100g基线） | name, unit, calorie, protein, fat, carb, sugar, sodium |
| `diet_record` | 饮食记录（含营养快照） | user_id, food_id, food_name, meal_type, amount, source, calorie, protein, fat, carb, sugar, sodium |
| `nutrition_record` | 每日营养汇总 | user_id, date, calorie_total, protein_total, score |
| `recipe` | 食谱库 | name, ingredients, steps, tags, calorie, protein |
| `recommendation` | 推荐记录与反馈 | user_id, recipe_id, reason, score, feedback |
| `alert_rule` | 预警阈值规则 | user_id, nutrient_type, threshold, enabled |

> **设计要点**：`diet_record` 表存储每条食物的6项营养快照（热量/蛋白质/脂肪/碳水/糖/钠），写入时优先使用 AI 识别返回的营养数据，兜底从 `food_item` 库按份量比例计算。每日营养汇总和评分直接 SUM `diet_record` 表的营养字段，不再 JOIN `food_item`。

### 初始化数据规模

- **食物库**：20 种常见食物（米饭、面条、鸡腿、鸡蛋、牛奶、西兰花、三文鱼、虾仁等），覆盖主食/肉类/蔬菜/水果/海鲜/乳制品/豆制品
- **食谱库**：10 道推荐食谱（鸡胸肉沙拉、番茄炒蛋、清蒸三文鱼、牛肉面等）
- **演示用户**：1 个（昵称"健康达人"，减脂目标）
- **演示记录**：6 条饮食记录（今日早餐+午餐）
- **默认预警**：3 条阈值规则

## 六、API 接口设计

遵循 RESTful 规范，所有接口统一返回 `ApiResponse<T>` 格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

| 接口路径 | 方法 | 功能 |
|---------|------|------|
| `/api/food/recognize` | POST | 上传图片 → 食物识别候选 |
| `/api/voice/parse` | POST | 上传语音 → 转写 + 食物实体 |
| `/api/diet-records` | POST | 新增饮食记录 |
| `/api/diet-records` | GET | 查询某日饮食记录列表 |
| `/api/diet-records/{id}` | PUT | 修改饮食记录 |
| `/api/diet-records/{id}` | DELETE | 删除饮食记录 |
| `/api/nutrition/daily` | GET | 查询营养汇总 + 趋势 |
| `/api/health-score/daily` | GET | 查询健康评分 + 建议 |
| `/api/recommendations/today` | GET | 获取今日推荐列表 |
| `/api/recommendations/feedback` | POST | 提交不喜欢反馈 + 刷新 |
| `/api/alert-rules` | POST | 创建预警规则 |
| `/api/alert-rules` | GET | 查询用户的预警规则 |
| `/api/alert-rules/{id}` | PUT | 修改预警规则 |
| `/api/alert-rules/check` | GET | 检测是否超标 |
| `/api/user-profile` | GET | 查询用户档案 |
| `/api/user-profile` | PUT | 更新用户档案 |

## 七、AI 能力集成

系统虽为课程演示项目，但设计了完整的 AI 适配层：

| AI 能力 | 适配器类 | 当前实现 | 设计说明 |
|---------|---------|---------|---------|
| 图像识别 | `ImageRecognitionAdapter` | 模拟返回 3 个候选 | 可对接 YOLO / 百度AI / 阿里云视觉 |
| 语音转文字 | `SpeechToTextAdapter` | 模拟返回预设文本 | 可对接讯飞 / Whisper / 阿里云语音 |
| 食物实体解析 | `FoodEntityParserAdapter` | 从文本解析食物+份量 | 可对接 NER 模型 |
| 消息推送 | `NotificationAdapter` | 模拟推送日志 | 可对接极光推送 / 站内信 |

**设计原则：** 适配器模式封装第三方差异，替换实现类即可切换 AI 服务提供商，业务层代码无需修改。

## 九、设计原则

| 原则 | 体现 |
|------|------|
| **模块化** | 按数据采集、营养分析、推荐、预警、档案拆分模块 |
| **单一职责** | Controller 处理请求，Service 处理业务，Repository 持久化 |
| **可替换** | AI 适配器封装，可切换不同服务商 |
| **用户可控** | AI 识别结果必须经用户确认才能落库，支持修改和删除 |
| **数据最小化** | 仅采集课程演示必要数据，不涉及实名/支付/医疗等高敏信息 |
| **配置化** | 评分规则、推荐参数、阈值优先配置化或表驱动 |
| **异常处理** | 统一异常处理 + 中文友好提示 |

## 十一、应用场景与价值

### 目标用户
- **普通饮食记录用户**：希望快速记录每天吃了什么
- **健康管理用户**：关注热量、营养均衡、目标完成度
- **慢性病/特殊饮食用户**：关注糖/钠/脂肪预警和忌口提醒

### 核心价值
1. **降低记录门槛**：拍照/语音代替手动输入，三步完成记录
2. **数据驱动洞察**：将饮食数据转化为可视化的营养分析和健康评分
3. **个性化体验**：根据目标、偏好、忌口定制推荐和建议
4. **风险主动预警**：超标时主动提醒，帮助用户及时调整

## 十二、局限性与未来展望

### 当前局限
- AI 识别为模拟实现，未接入真实第三方 API
- 无用户登录注册（演示环境固定用户 ID=1）
- 推荐算法为基于内容的简单评分，未使用协同过滤
- 无后台管理功能（食物库/食谱库固定初始化数据）

### 可扩展方向
- 接入真实 AI 识别服务（百度/阿里云视觉、讯飞语音）
- 增加用户认证（JWT + 登录注册）
- 推荐算法升级（协同过滤 + 矩阵分解）
- 增加社区功能（饮食打卡分享）
- 接入可穿戴设备数据
- 部署到云端服务器

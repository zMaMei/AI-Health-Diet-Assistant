# AI智能个人健康饮食助手 — 软件全面介绍

## 一、项目概述

**AI智能个人健康饮食助手**是一款面向普通用户的移动端健康饮食管理应用，作为杭州师范大学（HZNU）课程设计项目，由21小组4名成员共同完成。

系统通过**拍照识别、语音输入、手动编辑**三种方式采集用户的饮食数据，接入阿里云百炼（DashScope）通义千问多模态 AI 大模型自动估算食物营养成分，并根据用户个性化的健康目标（减脂、增肌、控糖等）生成**健康评分、改进建议、个性化食谱推荐和超标预警**。

## 三、技术架构

### 3.1 总体架构

系统采用**前后端分离的分层架构**：

```
移动端 UI（Vue 3 + Vite）
    ↓ RESTful JSON API
Spring Boot 后端
    ├── Controller 层    — 请求入口、参数校验、统一响应
    ├── Service 层       — 业务逻辑（记录、营养计算、评分、推荐、预警、AI 对话）
    ├── AI Adapter 层    — 适配第三方 AI 服务（6个适配器）
    └── Repository 层    — JPA 数据访问
    ↓
数据库（MySQL）
```

|------|------|

| 技术 | 用途 |
|------|------|
| Vue 3 (Composition API) | 前端框架 |
| Vite 8 | 构建工具 |

| Axios | HTTP 请求 |
| Cropper.js | 头像裁剪 |

### 3.4 部署架构

```
开发者电脑
  ├── 后端进程 (port 8080)
  │   └── Spring Boot + MySQL
  └── 前端进程 (port 5173)
      └── Vite Dev Server → 代理 /api → 后端
```

前端通过 Vite 的 proxy 配置将 `/api` 请求转发到后端 8080 端口，开发时无需配置 Nginx 等反向代理。

## 四、详细功能模块

系统共 **4 个前端页面**（记录、分析、推荐、我的），整合 **6 大功能模块**：

### 模块 1：用户认证

**注册与登录**
- 用户名+密码注册，BCrypt 哈希存储密码，注册成功自动登录并创建初始健康档案和三条默认预警规则（热量/糖分/钠）
- 用户名+密码登录，服务端返回 UUID Token，客户端 localStorage 持久化，重启 App 自动恢复登录态
- 登出需二次确认，清除服务端 Token 和本地状态

**头像管理**
- 点击头像弹出 Cropper.js 裁剪模态框，支持拖拽定位、滚轮缩放、重新选图
- Canvas 裁剪为 256×256 后上传至 `uploads/avatars/` 目录

**未登录保护**
- 未登录时"我的"页设置区隐藏、显示登录提示
- 其他页面操作按钮触发 toast 提示"请先在'我的'页面登录"
- API 返回 401 时自动登出并提示

### 模块 2：饮食数据采集（记录页）

三种数据采集入口：

**① 拍照识别食物**
- 调用浏览器摄像头拍照或选择图片，上传至后端
- 后端保存照片到 `uploads/diet-images/` 目录，同时调用通义千问多模态 API 识别食物
- AI 返回候选食物列表，每项含：食物名称、置信度、6项营养估算值（复合菜品如"韭菜炒鸡蛋"不拆分，作为一个整体识别）
- 用户勾选要保存的食物、调整份量、选择统一餐次后批量保存
- 照片通过 `meal_photo` 表按餐次关联，支持一餐多张照片横向滑动预览
- 低置信度或未识别时提示用户改用手动添加

**② 语音输入饮食记录**
- 调用浏览器麦克风实时录音（MediaRecorder API），显示录音计时
- 停止录音后自动上传至后端，保存 `.webm` 音频到 `uploads/voice/` 目录
- 后端调用通义千问 API 进行语音转文字 + 食物实体解析（含营养估算）
- 解析结果写入 `voice_record` 表（含音频路径、转写文本、食物实体 JSON）
- 前端展示与拍照模态一致的交互：勾选食物、查看营养、调整份量、选择统一餐次
- 语音记录关联到对应餐次卡片，支持 ▶️ 播放录音和查看转写文本

**③ 手动添加**
- 表单填写：食物名称、餐次（早餐/午餐/晚餐/夜宵/其他）、份量
- **必须点击「智能分析」**调用 AI 获取营养数据后，保存按钮才变为可点击状态
- 当 AI 识别失败或用户需要补记时使用

**记录管理（手风琴布局）：**
- 按餐次分组以手风琴卡片展示今日已记录的食物
- 折叠状态：显示餐次图标+名称、该餐热量/蛋白质合计
- 展开状态依次展示：
  1. **照片滑动区**：多张照片横向滑动，点击缩略图弹出全屏大图预览
  2. **语音记录区**：紫色卡片显示录音时长、转写文本、▶️ 播放按钮
  3. **食物列表**：每条食物显示名称、份量、来源标签、6项营养值、编辑/删除按钮
- 顶部今日总览卡片：展示6项营养当日总和（热量/蛋白质/脂肪/碳水/糖/钠）

### 模块 3：智能营养分析与健康评分（分析页）

本页面整合了营养分析、健康评分和 AI 饮食对话三大功能：

**日期导航：**
- 日期选择器支持前后切换和历史日期回顾（不可选未来日期），附「今天」快捷按钮

**当日饮食回顾：**
- 同记录页手风琴展示各餐次记录，包括照片轮播、语音播放、食物条目，支持编辑和删除

**健康评分：**
- SVG 环形进度图展示 0-100 评分
- 评分逻辑：基础分 80 分 + 热量超标扣分（每超出目标 10% 扣 1 分，最多扣 10 分）+ 糖分 > 阈值扣分 + 钠 > 阈值扣分 + 蛋白质达标加分
- 当日记录不足 2 餐 → 不评分，提示"记录更多饮食后可获得评分"
- ✅ 优点列表（热量控制良好/糖分控制良好等）
- ⚠️ 风险项列表（蛋白质摄入不足等）

**营养分析：**
- 热量/蛋白质/脂肪/碳水四大指标，展示摄入量、目标值、完成百分比环形进度
- 点击营养素查看当日该营养素的食物来源明细弹窗
- 近一周热量趋势柱状图（当日高亮），便于观察饮食趋势
- 自动生成饮食改进建议

**目标体系：**

| 目标 | 热量 | 蛋白质 | 脂肪 | 碳水 |
|------|------|--------|------|------|
| 均衡 | 2000 | 60g | 65g | 300g |
| 减脂 | 1600 | 70g | 50g | 200g |
| 增肌 | 2500 | 120g | 70g | 350g |

**AI 饮食对话：**
- 以弹窗形式展示 AI 对话界面，支持与通义千问进行当日饮食分析对话
- 预设「开始分析」按钮发送初始分析提示（自动附带用户档案、近 7 天饮食记录和营养汇总作为上下文）
- 支持自由提问、多轮对话，消息历史持久化到 `ai_conversation` 和 `ai_message` 表
- 支持简单 Markdown 渲染，消息区域自动滚动至底部

### 模块 4：个性化食谱推荐（推荐页）

**推荐机制（AI 优先，规则引擎降级）：**

1. **AI 推荐**（优先）：调用通义千问多模态 API，根据用户营养缺口、目标、口味偏好、忌口，从菜谱库中选择 5 道最合适的菜谱并生成推荐理由，或直接创造全新菜谱
2. **规则引擎**（降级，AI 不可用时自动启用）：
   - 过滤阶段：排除用户忌口的食谱、历史"不喜欢"的食谱
   - 评分阶段（0-100分）：基础分 50 + 目标匹配（±20分）+ 口味偏好匹配（+5/项）+ 标签匹配（+15分）
   - 高分段随机选取 5 道，增加多样性

**推荐理由生成：**
AI 根据用户的实际营养缺口和偏好自动生成个性化推荐理由，如：
> "推荐理由：今日蛋白质摄入不足，这道菜高蛋白低脂肪，适合减脂目标"

**交互功能：**
- 推荐卡片展示：菜名、标签、营养对比条（热量/蛋白质/脂肪/碳水 vs 用户阈值）、食材预览、推荐理由
- **换一批**：调用 AI 直接创造全新菜谱（不限菜谱库），带旋转加载动画
- **查看详情弹窗**：完整营养数值（含糖/钠）、完整食材列表、分步骤做法、推荐理由
- 营养对比条使用后端返回的用户阈值，自动适配个性化目标

### 模块 5：饮食提醒与预警

**预警检测时机：** 每次查看推荐页或调用 `/api/alert-rules/check` 时自动检测

**检测逻辑：**
```
当日累计摄入 > 用户设置的阈值 → 生成预警消息
```

**预警规则管理：**
- 支持 6 项营养素独立设置：热量、糖分、钠、蛋白质、脂肪、碳水
- 每项独立开关，修改数值自动保存
- 注册时自动创建热量/糖分/钠三条默认预警规则

**AI 智能阈值分析：**
- 「AI 智能分析」按钮：根据用户年龄、性别、身高、体重、BMI、健康目标、慢性病等信息
- 调用通义千问生成个性化营养摄入上限建议，自动更新预警规则

### 模块 6：健康档案与设置（我的页）

- **个人资料**：头像（可裁剪上传）、昵称（点击编辑）、年龄、身高、体重、性别
- **健康目标**：均衡饮食 / 减脂 / 增肌 / 控糖
- **口味偏好**：多选（清淡、中式、西式、日式、辣味、酸甜、咸鲜）+ 自定义标签
- **忌口设置**：多选（海鲜、花生、牛奶、鸡蛋、豆制品、辛辣、芝麻、芒果、酒精等）+ 自定义标签，支持行内编辑和删除
- **慢性病/特殊饮食标签**：糖尿病、高血压、高血脂、痛风 + 自定义标签
- **预警阈值**：6 项营养素独立开关和数值设置，支持 AI 智能分析一键生成
- 所有设置保存后实时影响推荐算法和预警检测

## 五、数据库设计

系统共设计 **12 张核心数据表**：

| 表名 | 说明 | 核心字段 |
|------|------|---------|
| `users` | 用户基础信息（含登录凭据） | id, username (唯一索引), nickname, password_hash, created_at |
| `user_profile` | 用户健康档案 | user_id, age, height_cm, weight_kg, goal, taboo, taste_preference, warning_profile, gender, avatar_url |
| `food_item` | 食物营养成分库（每100g基线） | name, category, unit, calorie, protein, fat, carbohydrate, sugar, sodium |
| `diet_record` | 饮食记录（含营养快照） | user_id, food_id, food_name, meal_type, amount, source, calorie, protein, fat, carbohydrate, sugar, sodium |
| `nutrition_record` | 每日营养汇总 | user_id, record_date, calorie_total, protein_total, fat_total, carbohydrate_total, sugar_total, sodium_total, score |
| `meal_photo` | 餐次照片（一餐多张） | user_id, record_date, meal_type, image_url |
| `voice_record` | 语音录音记录 | user_id, record_date, meal_type, audio_url, transcribed_text, food_entities (JSON), duration_seconds |
| `recipe` | 食谱库 | name, ingredients, steps, tags, calorie, protein, fat, carbohydrate, sugar, sodium |
| `recommendation` | 推荐记录 | user_id, recipe_id, reason, score, created_at |
| `alert_rule` | 预警阈值规则 | user_id, nutrient_type, threshold, enabled |
| `ai_conversation` | AI 对话会话 | user_id, record_date, created_at |
| `ai_message` | AI 对话消息 | conversation_id, role (USER/AI), content, created_at |

> **设计要点**：
> - `users` 含 `username`（唯一索引）和 `password_hash`（BCrypt 哈希），支持多用户注册登录。Token 为 UUID 字符串，服务端 `ConcurrentHashMap` 管理。
> - `user_profile` 含 `avatar_url`，头像通过裁剪模态框（Cropper.js）处理后上传至本地 `uploads/avatars/`。
> - `diet_record` 存储每条食物的 6 项营养快照，写入时 **AI 返回值为准**，`food_item` 仅兜底。每日营养汇总和评分直接 SUM `diet_record` 的营养字段。
> - `meal_photo` 按 `(user_id, record_date, meal_type)` 关联某一餐，一餐可有多张照片。
> - `voice_record` 存储录音文件路径、转写文本、AI 解析出的食物实体（JSON）。`meal_type` 在用户确认保存后回填，实现语音与餐次卡片的关联。
> - `ai_conversation` 和 `ai_message` 记录 AI 饮食对话历史，按日期组织会话，支持多轮对话。
> - 上传文件组织为 `uploads/{userId}/{YYYY}/{MM}/{DD}/{UUID}.ext`，便于按用户/日期管理和清理。

### 初始化数据规模

- **食物库**：20 种常见食物（米饭、面条、鸡腿、鸡蛋、牛奶、西兰花、三文鱼、虾仁等），覆盖主食/肉类/蔬菜/水果/海鲜/乳制品/豆制品
- **食谱库**：65 道一人食菜谱（覆盖减脂轻食、高蛋白增肌、快手小炒、汤粥主食、早餐简餐五大类）
- **演示用户**：1 个（用户名 `demo`、密码 `demo123`，昵称"健康达人"，减脂目标，BCrypt 哈希存储）
- **演示记录**：今日早餐+午餐共若干条饮食记录
- **默认预警**：3 条阈值规则（热量/糖分/钠）

## 六、API 接口设计

遵循 RESTful 规范，所有接口统一返回 `ApiResponse<T>` 格式：

```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

### 6.1 认证 Auth（4 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|
| `/api/auth/register` | POST | — | 注册新用户（自动登录，返回 token） |
| `/api/auth/login` | POST | — | 用户名+密码登录，返回 token |
| `/api/auth/logout` | POST | Bearer | 退出登录，服务端清除 token |
| `/api/auth/avatar` | POST | Bearer | 上传/更新头像（multipart） |

### 6.2 饮食记录 Diet Records（4 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|
| `/api/diet-records` | POST | Bearer | 新增饮食记录（AI 营养值优先，food_item 兜底） |
| `/api/diet-records` | GET | Bearer | 查询某日饮食记录列表 |
| `/api/diet-records/{id}` | PUT | Bearer | 修改饮食记录（校验所有权） |
| `/api/diet-records/{id}` | DELETE | Bearer | 删除饮食记录（校验所有权） |

### 6.3 食物识别 Food（2 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|
| `/api/food/recognize` | POST | Bearer | 上传图片 → 食物识别候选（含营养估算） |
| `/api/food/analyze-text` | POST | Bearer | 食物名称文本分析 → AI 估算营养（手动添加"智能分析"） |

### 6.4 语音解析 Voice（4 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|
| `/api/voice/parse` | POST | Bearer | 上传录音 → 保存文件 → ASR → 实体解析 → 返回结果 |
| `/api/voice-records` | GET | Bearer | 查询某日语音记录列表 |
| `/api/voice-records/{id}/meal-type` | PUT | Bearer | 回填语音记录的餐次类型（校验所有权） |
| `/api/voice-records/{id}` | DELETE | Bearer | 删除语音记录（记录+音频文件，校验所有权） |

### 6.5 餐次照片 Meal Photos（3 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|
| `/api/meal-photos` | POST | Bearer | 新增餐次照片记录（临时目录→最终目录） |
| `/api/meal-photos` | GET | Bearer | 查询某日/某餐次照片 |
| `/api/meal-photos/{id}` | DELETE | Bearer | 删除照片（记录+文件，校验所有权） |

### 6.6 营养分析 Nutrition（1 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|
| `/api/nutrition/daily` | GET | Bearer | 查询营养汇总 + 周趋势 + 建议 |

### 6.7 健康评分 Health Score（1 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|
| `/api/health-score/daily` | GET | Bearer | 查询健康评分（0-100）+ 优缺点 + 风险建议 + 周趋势 |

### 6.8 AI 饮食对话 AI Analysis（2 个接口）

| 接口路径 | 方法 | 认证 | 功能 |
|---------|------|------|------|

## 七、AI 能力集成

系统已接入**阿里云百炼（DashScope）**平台，通过 `qwen-omni-turbo` 多模态模型提供真实 AI 能力：

| AI 能力 | 适配器类 | API 端点 | 说明 |
|---------|---------|---------|------|
| 图像识别 | `ImageRecognitionAdapter` | 多模态生成 API | 传入图片 → AI 识别食物名称 + 估算 6 项营养成分（复合菜品不拆分） |
| 食物名称分析 | `ImageRecognitionAdapter.analyzeFoodByName()` | 多模态生成 API | 传入食物名 → AI 估算营养（手动添加的"智能分析"） |
| 语音转文字 | `SpeechToTextAdapter` | 多模态生成 API | 传入 .webm 音频 → qwen-omni-turbo 转写为中文文本 |
| 食物实体解析 | `FoodEntityParserAdapter` | 多模态生成 API | 传入文本 → AI 提取食物实体 + 估算营养成分 |
| 食谱推荐 | `RecommendationAdapter` | 多模态生成 API | 根据营养缺口/目标/偏好 → 从库中选最佳 5 道或直接创造全新菜谱 |
| 阈值分析 | `ThresholdAnalysisAdapter` | 多模态生成 API | 根据用户档案（年龄/BMI/目标/慢性病）→ AI 生成个性化营养摄入上限 |
| 消息推送 | `NotificationAdapter` | — | 预警消息推送（当前模拟实现） |

**Prompt 设计要点：**
- 严格的 JSON 返回格式约束，后端解析 AI 响应提取结构化数据
- **复合菜品不拆分**：明确要求"韭菜炒鸡蛋"等菜品作为一个完整食物识别
- 营养值均由 AI 根据实际内容估算，不依赖本地 `food_item` 库的固定值
- `food_item` 库仅补全 unit 和 category 字段，作为 AI 失败时的兜底

**降级策略：** AI 不可用时自动降级——图像/语音识别返回提示引导用户手动添加；推荐降级为规则引擎评分选取；阈值分析降级为默认推荐值。

**设计原则：** 适配器模式封装第三方差异，替换实现类即可切换 AI 服务提供商，业务层代码无需修改。

## 九、设计原则

| 原则 | 体现 |
|------|------|
| **模块化** | 按数据采集、营养分析、推荐、预警、档案拆分功能模块 |
| **单一职责** | Controller 处理请求，Service 处理业务，Repository 持久化，Adapter 封装 AI |
| **可替换** | AI 适配器封装，可切换不同服务商，业务层代码无需修改 |
| **用户可控** | AI 识别结果必须经用户确认才能落库，支持修改和删除 |
| **数据最小化** | 仅采集课程演示必要数据，不涉及实名/支付/医疗等高敏信息 |
| **降级保护** | AI 不可用时自动降级为规则引擎或本地兜底，保证服务可用 |
| **所有权校验** | 所有涉及用户数据的修改/删除操作均验证数据归属，防止越权 |
| **异常处理** | 统一异常处理 + 中文友好提示 |

## 十一、应用场景与价值

### 目标用户
- **普通饮食记录用户**：希望快速记录每天吃了什么
- **健康管理用户**：关注热量、营养均衡、目标完成度
- **慢性病/特殊饮食用户**：关注糖/钠/脂肪预警和忌口提醒

### 核心价值
1. **降低记录门槛**：拍照/语音代替手动输入，三步完成记录
2. **数据驱动洞察**：将饮食数据转化为可视化的营养分析和健康评分
3. **个性化体验**：根据目标、偏好、忌口定制推荐和建议
4. **AI 深度对话**：基于当日饮食数据的专业分析反馈，帮助用户养成更好的饮食习惯
5. **风险主动预警**：超标时主动提醒，帮助用户及时调整

## 十二、局限性与未来展望

### 当前局限
- Token 存储在内存（服务重启后所有用户需重新登录）
- 推荐算法为基于内容的评分+AI增强，未使用协同过滤
- 无后台管理功能（食物库/食谱库固定初始化数据）
- 头像上传路径依赖 JVM 工作目录（生产环境应配置绝对路径）

### 可扩展方向
- Token 持久化（Redis / 数据库）或升级为 JWT
- 推荐算法升级（协同过滤 + 深度学习）
- 增加社区功能（饮食打卡分享）
- 接入可穿戴设备数据（Apple Health / 华为健康）
- 部署到云端服务器
